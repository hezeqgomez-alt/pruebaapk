Arquitectura y calidad de código

✅ Fortalezas

  • Separación de responsabilidades clara en la mayoría de los componentes
  • Uso correcto de Context API para autenticación (AuthContext)
  • Patrón defensivo con ErrorBoundary en secciones críticas
  • Manejo de cuota de localStorage con degradación graceful

⚠️ Problemas críticos

A. App.jsx monolítico (1.031 líneas) HIGH
src/App.jsx · líneas 88–550

Un solo componente maneja: transacciones + sincronización cloud, cola de toasts (hasta 4 simultáneos), 8+ modales, progreso de OCR, tracking de PDFs, modo oscuro, presupuestos y categorías personalizadas.

Riesgo: Imposible hacer unit testing sin mockear 15+ hooks. Cualquier bug en estado puede afectar funciones no relacionadas.

Refactor sugerido: Extraer en hooks personalizados:

  • useTransactionManager() → estado + sincronización cloud
  • useToastQueue() → cola de notificaciones
  • useModalStack() → unificar 8 estados de modal en uno solo
  • usePdfImport() → manejo de archivos + progreso OCR
B. Acoplamiento en pdfParser.js MEDIUM
src/utils/pdfParser.js · líneas 474–679

parseRows() y parseColumnar() tienen +150 líneas de lógica casi idéntica: extracción de tarjeta, parseo de fechas, detección de cuotas, sanitización de descripciones.

Fix: Extraer extractTransaction(row, refYear, currentCard) → retorna { date, desc, amount, installment, fx }

C. Función fmt() duplicada en 6 archivos MEDIUM

La misma función de formato de moneda (Intl.NumberFormat es-AR ARS) está copiada en StatsCards, BudgetPanel, InsightsPanel, InstallmentsPanel, LoansPanel y reportGenerator.js.

Fix: Crear src/utils/format.js con export const ARS = (n) => ...

2
Performance
A. Re-renders innecesarios HIGH
src/App.jsx · líneas 885, 932

StatsCards y TransactionList re-renderizan con cualquier cambio de estado en App.jsx (incluso darkMode o apertura de modales), porque no están memoizados.

filteredForReport actualiza en cada cambio de filtro en TransactionList → dispara re-render en App → re-renderiza BudgetPanel, InstallmentsPanel, etc.

Fix: Envolver StatsCards y TransactionList con React.memo(). Pasar solo los datos que cada componente necesita.

B. detectUnnecessary() es O(n×m) en cada render HIGH
src/utils/categorizer.js · líneas 421–561

La función normaliza la descripción de cada transacción para cada comercio único. Con 5.000 transacciones puede generar 500.000 llamadas a normalize() por re-render.

Fix: Cachear las descripciones normalizadas en el objeto de transacción al momento del parseo, o memoizar el mapa de normalización:

// en App.jsx
const normCache = useMemo(
  () => new Map(transactions.map(t => [t.id, normalize(t.description)])),
  [transactions]
)
C. Worker OCR sin manejo de crash MEDIUM
src/utils/pdfParser.js · líneas 771–826

El worker OCR es un singleton global. Si crashea (memoria insuficiente en PDF grande), todos los uploads pendientes fallan silenciosamente sin posibilidad de retry.

Fix: Detectar crash del worker y reiniciarlo; agregar lógica de retry con backoff.

D. Bundle innecesariamente grande en primer load LOW

pdfjs-dist y tesseract.js (5MB WASM) se cargan aunque el usuario nunca suba un PDF escaneado.

Fix: Dynamic import — cargar el módulo OCR solo cuando se detecta un PDF escaneado.

Gaps de UX y producto
A. Cuotas: no hay estado para pagadas anticipadamente MEDIUM
src/components/InstallmentsPanel.jsx · líneas 48–65

Las cuotas solo tienen estado "activa". Si el usuario cancela una compra en cuotas, refinancia o paga anticipado, la proyección de los próximos meses queda incorrecta.

Fix: Agregar campo status: 'active' | 'paused' | 'prepaid' con UI para marcar manualmente.

B. Presupuestos sin carry-over mensual MEDIUM
src/components/BudgetPanel.jsx

Si el usuario configura $50.000 para alimentación y gasta $30.000, los $20.000 no pasan al mes siguiente. Los presupuestos son planos y no reflejan el comportamiento real del gasto.

Fix: Agregar opción rollover: true/false por categoría. Calcular adjustedLimit = limit + underspend_prev_month.

C. Informe PDF sin proyección de cuotas futuras LOW

InstallmentsPanel proyecta los próximos 3 meses pero esa información no aparece en el informe PDF exportado. El informe está incompleto para usuarios con muchas cuotas.

D. Undo sin contador regresivo LOW
src/components/TransactionList.jsx · línea 349

El botón "Deshacer" desaparece silenciosamente a los 5 segundos. El usuario no sabe que pierde esa oportunidad.

Fix: Mostrar "Deshacer (3s)" y cambiar color del botón a medida que se acerca el vencimiento.

E. Sin flujo para fusionar transacciones duplicadas MEDIUM

La deduplicación al subir PDFs usa match exacto de fecha|monto|descripción|origen. Variaciones mínimas en la descripción ("NETFLIX" vs "NETFLIX.COM") resultan en duplicados que el usuario no puede resolver fácilmente.

4
Bugs y casos borde
A. Merge cloud/local falla con descripciones alteradas HIGH
src/App.jsx · líneas 176–230

La clave de merge es fecha|monto|descripción|origen. Si OCR normaliza "MCDONALD" como "MCDONALDS" en un segundo parse, o si el usuario edita la descripción, la misma transacción se duplica en el siguiente sync.

Fix crítico: Usar el campo id (UUID) como clave de merge — ya se genera con randomUUID() al parsear. Requiere asegurar que los IDs persisten en el round-trip localStorage ↔ Supabase.

B. Cuotas agrupadas por descripción colapsan compras iguales HIGH
src/components/InstallmentsPanel.jsx · líneas 20–45src/utils/exportXLSX.js · líneas 111–137

Si el usuario compró el mismo artículo dos veces en el mismo comercio con cuotas idénticas (ej: dos portátiles $300.000 en 12 cuotas), ambas se colapsan en una sola entrada mostrando el doble del monto.

Fix: Agrupar por id + descripción en lugar de solo descripción.

C. Race condition: cloud save después de logout MEDIUM
src/App.jsx · líneas 233–254

El debounce de 2s puede dispararse después de que el usuario cierre sesión. El timer sigue activo con el user.id del estado anterior.

Fix: Cancelar el timer pendiente cuando user cambia a null.

useEffect(() => {
  return () => { if (debounceTimer.current) clearTimeout(debounceTimer.current) }
}, [user])
D. OCR progress se queda colgado si el componente se desmonta LOW

Si el usuario navega mientras un PDF se procesa con OCR, el worker sigue corriendo pero el componente ya no puede actualizar el progreso. No hay cleanup al desmontar.

5
Seguridad
✅ CSV/Excel injection correctamente protegido
src/utils/exportXLSX.js · líneas 13–16

La función sanitizeCell() prefija con ' cualquier celda que empiece con =, +, -, @. Está aplicada en todos los campos. Bien implementado.

A. Sin sanitización HTML en notas de transacción LOW
src/components/TransactionList.jsx · líneas 103–116

Las notas se muestran como texto plano hoy — sin riesgo inmediato. Pero si en el futuro se decide renderizarlas como HTML (markdown, links), habría riesgo XSS. Sanitizar preventivamente al guardar.

B. Caracteres de control en texto OCR no sanitizados LOW
src/utils/pdfParser.js · línea 832

El texto extraído por OCR puede contener caracteres de control (\x00–\x1f) que podrían romper el render o exportaciones.

Fix: line.replace(/[\x00-\x1f]/g, ' ').trim() antes de procesar.

C. Electron: URL de apertura externa sin whitelist LOW
src/components/LicenseGate.jsx · línea 109

window.electronAPI.openExternal(url) sin validar el dominio destino. Riesgo bajo porque la URL está hardcodeada, pero si se parametriza en el futuro es un vector.

Experiencia mobile
A. Touch targets demasiado pequeños HIGH
src/components/TransactionList.jsx · línea 641

El botón eliminar individual es w-8 h-8 (32px). El estándar de accesibilidad WCAG es 44px mínimo. En mobile con dedos normales el tap falla con frecuencia.

Fix inmediato: Cambiar a w-10 h-10 -m-1 en todos los botones de icono de la lista.

B. Modales no cierran con swipe hacia abajo MEDIUM

En iOS y Android el gesto esperado para cerrar un modal es swipe down. Actualmente solo funciona click en el backdrop o botón X. Causa fricción notable en uso cotidiano.

Fix: Agregar handler de gesto con useGesture o Framer Motion drag constraints.

C. Sin indicador de modo offline LOW

Si el usuario está sin conexión, el cloud sync falla silenciosamente (solo console.warn). Los datos están seguros en localStorage pero el usuario no lo sabe — puede generar desconfianza.

Fix: Mostrar badge "Sin conexión · datos guardados localmente" cuando la sync falla reiteradamente.

D. Filtros de fecha se superponen en pantallas muy pequeñas LOW
src/components/TransactionList.jsx · líneas 460–466

Los dos date pickers están en la misma fila. En pantallas <320px se comprimen y son inutilizables.

Monetización y crecimiento
A. Trial demasiado generoso — fuga de revenue HIGH

El trial de 30 días no tiene tope de PDFs (se eliminó intencionalmente). Un usuario puede subir 100 PDFs durante el trial, extraer todo el valor de la app, y no convertir.

Fix sugerido: Re-introducir un tope suave que se vaya liberando: 3 PDFs en la primera semana, +2 por semana. Así el usuario ve valor progresivo y siente urgencia de convertir.

B. Sin funnel de conversión dentro de la app HIGH

Los usuarios de trial ven el banner solo cuando los días son pocos. No hay prompts contextuales al usar features premium.

Momentos de conversión no aprovechados:

  • Después del 3er informe PDF generado
  • Al llegar a 50+ transacciones cargadas
  • Al usar el panel de Préstamos por primera vez
  • Al exportar a Excel por segunda vez

Fix: Mostrar modal de upgrade contextual en esos momentos con el mensaje apropiado.

C. Diferenciación Pro vs. Free no está comunicada MEDIUM

No hay tabla de comparación de planes. El usuario que entra al sitio no entiende qué gana pagando. Todo parece gratuito hasta que expira.

Fix: Agregar sección "¿Qué incluye Pro?" en la pantalla de expiración y en el drawer mobile.

D. Sin mecanismo de referidos LOW

El botón "Compartir" genera un texto genérico sin tracking. No hay incentivo para que el usuario actual refiera nuevos usuarios.

Fix: Generar código de referido por usuario. "Compartí XXXX — los dos ganan 7 días gratis."

E. Features premium no implementados aún MEDIUM

Oportunidades claras para el tier Pro que aún no existen:

  • Informe PDF mensual automático por email
  • Link de reporte compartible (solo lectura) para contador/familia
  • Alertas por exceso de presupuesto via push/email
  • Exportación bulk + programada

Oportunidades de simplificación
A. Consolidar 8 estados de modal en uno solo
src/App.jsx · líneas 114–120
// Antes (5 booleans independientes):
const [showAddModal, setShowAddModal] = useState(false)
const [showBankGuide, setShowBankGuide] = useState(false)
...

// Después (1 estado con string):

Los dos parsers (parseRows y parseColumnar) comparten ~150 líneas de lógica. Extrayendo extractTransaction() se eliminarían ~80 líneas de código duplicado y se facilitaría agregar nuevos bancos.

La lógica Electron (window.electronAPI) está dispersa en 6+ lugares de App.jsx. Extraer useElectronLicense() mejoraría legibilidad y facilitaría el mantenimiento del build de escritorio.

src/utils/pdfParser.js · líneas 772–774

La variable _ocrWorkerReady se setea pero nunca se lee. Se puede eliminar y usar _ocrWorker !== null directamente.

Resumen ejecutivo

La app está bien construida para su etapa. El parseo de PDFs es el componente más sofisticado y funciona correctamente para los principales bancos argentinos. Los mayores riesgos actuales son dos bugs de datos silenciosos (merge por descripción, cuotas colapsadas) y una fuga de revenue clara (trial sin tope + sin funnel de conversión).

La deuda técnica (App.jsx monolítico, fmt() duplicado) no bloquea el producto hoy pero ralentizará el desarrollo en 2–3 meses si no se aborda. Las mayores oportunidades de crecimiento están en monetización (Fase 3) y en features Pro diferenciados (Fase 4).
